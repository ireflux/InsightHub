## 长上下文不再奢侈，控制权争夺蔓延至硬件与法律

Anthropic 本周宣布，Claude Opus 4.6 与 Sonnet 4.6 的 1M 上下文窗口正式全面可用，且**标准定价覆盖全窗口，无长上下文溢价**——此前同类功能常以倍率计价。这意味着一次 90 万 token 的请求与 9 千 token 的请求单价完全相同，同时媒体处理上限从 100 张/页提升至 600 张/页。变化看似技术细节，实则重塑了 AI 应用的经济模型：长上下文从此不再是“实验性奢侈功能”，而成为可计费、可规划的基础单元。

与此同时，一场关于“计算控制权”的争夺，正以意想不到的形式同时在法律与硬件层上演。蒙大拿州刚通过的《Right to Compute Act》名称为用户赋予计算自由，但技术社区讨论迅速揭示其潜在限制——法案可能禁止远程证明等技术，反而削弱用户对自身设备的控制。几乎同一时间，硬件黑客 Andrew “bunnie” Huang 推出的 Baochip-1x 微控制器，却选择在低性能 MCU 类别中罕见地集成 MMU（内存管理单元），从硬件层为用户提供运行不可信代码、隔离应用的能力。两者看似无关，却共同指向一个趋势：当 AI 代理需要持久上下文与复杂运行时，**“谁有权决定代码如何运行”这一根本问题，正从软件协议下沉到芯片设计，并向上延伸至法律文本**。

---

### 一、AI 上下文经济化：从营销噱头到计价单位

Claude 的更新核心在于定价结构的根本改变。官方公告明确写道：“Standard pricing now applies across the full 1M window for both models, with no long-context premium.”（标准定价现适用于两个模型的完整 1M 窗口，无长上下文溢价。）这直接否定了行业常见的“长上下文倍数收费”模式。结合媒体限制扩大至 600 张图片或 PDF 页，**1M 上下文从一个需要权衡成本的高级功能，转变为可无差别使用的标准资源**。

这一变化的产业 implication 是深远的。AI 应用设计长期以来受限于上下文窗口成本：开发者被迫进行手动上下文压缩、分段处理，或干脆放弃长文档分析。当 1M 上下文按标准 token 计价，且保持全速率限制时，上述约束被大幅放宽。企业可更自由地构建需要完整记忆的 AI 代理、分析整本合同或代码库，或维持数小时的高质量对话连续性。**心理门槛的降低可能比纯粹的技术参数更重要**——用户不再需要每次请求前精打细算“值不值得用长上下文”。

社区讨论清晰呈现了需求的两极。一位长期用户坦言：“I’ve never needed 1M, or even 250k+ context. I’m usually under 100k per request.”（我从不需要 1M，甚至 250K+，通常单次请求低于 100K。）而另一位开发者则视此为“huge”改进，尤其对 Claude Code 用户而言，前提是模型在超长上下文中的一致性得以保持。这种分歧本身是关键信号：**长上下文的真正价值，取决于具体工作流能否利用其规模，而非单纯追求数字本身**。

下一步需观察三点：第一，OpenAI、Google 等竞争对手是否会调整长上下文定价策略，避免在成本上处于劣势；第二，开发者是否真将 1M 窗口纳入常规设计，还是仅作为“备用选项”；第三，模型在 500K-1M 区间的性能衰减（coherence）是否会成为新的瓶颈——如一位用户提到，此前因“compaction”导致模型遗忘关键信息而切换至其他模型，此次回归取决于一致性是否改善。

### 二、控制权新战场：从 Montana 法案到 Baochip 的 MMU

**Montana 的《Right to Compute Act》** 在 Hacker News 引发的讨论迅速偏离了法案标题的乐观叙事。标题宣称“保障计算权”，但评论区普遍质疑其实际内容。用户 lukeschlather 指出：“it really looks like it simply prevents ‘the government’ from regulating the right to ‘make use of computational resources.’”（它似乎只是阻止政府监管‘使用计算资源’的权利。）更尖锐的是 matheusmoreira 的批评：真正的“计算权”法律应“ban remote attestation, force companies to allow custom software”（禁止远程证明，强制企业允许自定义软件），但该法案似乎并未触及这些核心管控点。Avicebron 更是直接援引支持声明的后半句：“This bill will help position Montana as a world-class destination for AI and Data Center investment.”（该法案将帮助蒙大拿成为 AI 与数据中心投资的世界级目的地。）**产业游说动机明显——通过限制政府监管能力，为大型数据中心铺设更宽松的合规环境，而非直接扩大用户对设备的控制**。法案的实质可能是一种“管制俘获”，以权利之名，行吸引投资之实。

**Baochip-1x 则从硬件层提供了另一种思路**。其设计者 bunnie 在项目更新中强调，该 RISC-V 微控制器的关键差异化特征是“the inclusion of a Memory Management Unit (MMU)”——这是同类性能/集成度芯片中罕见的配置。MMU 的核心价值在于提供页式虚拟内存，使每个应用运行在独立的地址空间，从而**允许用户安全地加载、运行不可信代码，而无需担心其破坏系统或窃取数据**。bunnie 将其与 CHERI、PMP 等技术对比，明确表示 MMU 是“composable”且“pragmatic”的选择：它经受数十年安全验证，且与其他保护机制可共存。硬件层面的 MMU，直接赋予了用户（而不仅是设备制造商）“运行任意应用”的能力——这与智能手机的封闭生态形成鲜明对比。

两者看似一法律一硬件，实则共同指向 **“控制权”定义的争夺**。Montana 法案试图在政府与企业间重新划分监管权责，但可能无意中削弱用户对个人设备的技术控制（例如，若远程证明被禁，用户更难验证固件完整性）；Baochip 则从硅片层面将控制权交还给运行代码的用户。当 AI 代理需要深入操作系统、甚至直接操作硬件时，**底层硬件是否提供 MMU 这类基础隔离能力，将决定用户能否安全地运行第三方 AI 服务，还是被锁定在厂商提供的封闭环境中**。

接下来值得盯住：其他州是否提出定义更清晰、真正限制厂商锁定的“计算权”法案；主流 RISC-V MCU 厂商（如 Espressif、-Wide 等）是否会跟进 MMU 设计；安全社区对远程证明与设备控制辩论的进一步深化。硬件层的开放控制能力，可能比法律条文更直接地塑造未来 AI 终端的形态。

### 三、技术深潜：Python 的永恒权衡与优化阶梯

在 soft 层面，开发者的日常则被另一种永恒的权衡占据：Python 的动态性与其性能缺陷。一份详尽的 benchmark 分析（ reproduced from the Benchmarks Game）量化了这一困境：在纯计算任务中，CPython 3.13 比 C 慢 20 倍至 875 倍。但作者的核心观点并非“Python 慢”，而是**“慢的原因与代价”**——Python 设计保证“任何对象可在运行时被修改”，导致每个 `a + b` 操作都必须经过复杂的类型检查与分发，而 C 的一条整数加法仅需一条 CPU 指令。对象内存开销（28 字节 vs 4 字节）是这一动态性的直接代价。

文章由此提出“优化阶梯”框架：开发者可根据投入成本，逐级选择工具——从保持纯 Python（依赖 NumPy/Pandas 等 C 扩展）到 Numba（JIT 编译热点循环）、Cython、Rust（via PyO3）乃至 Mojo。评论补充了重要维度：`__mharrison__` 指出 GPU 加速常被忽略，“Some numpy and pandas code can get a big speedup by using GPUs (same code with import change)”；`adsharma` 则提出“transpile to rust pyO3”可能位于阶梯顶端。而 CPython 3.13 的实验性“copy-and-patch JIT”与 3.15 计划融合 PyPy 追踪技术，暗示官方正在系统性地缓解动态性开销。

对泛科技读者而言，这一分析的价值在于**破除单一技术神话**：没有“银弹”能让 Python 既保持完全动态性又达到 C 的速度。工程决策的本质是权衡：数据科学工作流停留在 NumPy 沙盒已足够；高频交易内核则可能需 Rust 重写。理解“动态性开销发生在哪里”（如 `@cython.cdivision(True)` 消除除零检查的分支预测损失），比盲目追求最新工具更重要。

### 雷达：值得关注的开发者项目

- **Megadev**：为 Sega Mega Drive/Mega CD 提供的现代 C/68k 开发框架。其价值不在商业规模，而在**维护经典平台的可编程性**，满足复古游戏开发、嵌入式教育或硬件黑客需求。如果你对 16 位时代架构或特定芯片编程有好奇，这是一个低门槛的切入点。  
- **Baochip-1x**：已作为主线讨论。其开源许可细节在评论区引发疑问（vintagedave 指出信息不够透明），但核心在于它提供了一个**结合 MMU 与开放 RTL 的实用案例**，适合对高保证嵌入式系统或 RISC-V 安全架构感兴趣者跟踪。

### 接下来要盯的信号

1. **Claude 1M 上下文的实际采用数据**：Anthropic 是否公布长请求比例？开发者工作流是否真向 1M 迁移？  
2. **竞争对手定价反应**：OpenAI GPT-4.5/5、Google Gemini 是否调整长上下文溢价策略？  
3. **Montana 法案的后续立法与司法挑战**：其他州是否复制？是否有团体起诉其与“用户控制权”的潜在冲突？  
4. **MCU 市场的 MMU 趋势**：除 Baochip 外，是否有主流厂商（如 ST、Espressif）宣布类似产品？  
5. **Python JIT 在真实项目的性能报告**：CPython 3.13/3.15 的 JIT 在 Web 服务、数据管道中的实际收益是否显著？  
6. **远程证明（Remote Attestation）在州法案中的定位**：它会被明确禁止，还是作为“用户权利”的例外被保留？

---

控制权议题将持续发酵。当 AI 代理需要跨越多层软件栈、甚至直接操作硬件时，芯片是否提供隔离能力、法律是否允许用户验证运行环境，将比模型参数大小更具根本性。同时，长上下文定价的平价化，标志着生成式 AI 正从“演示驱动”转向“成本驱动”的实用阶段——**真正的竞争不再是谁的窗口更长，而是谁能用更低的边际成本，构建需要持久记忆的复杂应用**。
